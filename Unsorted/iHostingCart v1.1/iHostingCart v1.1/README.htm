1. INTRODUCTION

Congratulations on purchasing iHostingCart, the most powerful domain and hosting shopping cart solution on the market. We've designed iHostingCart with both you and your customer in mind. You'll find it easy to install and configure, whilst your customers will benefit from a smoother domain and hosting shopping experience. Just follow the simple instructions below. If you have any difficulties, make sure you check out our on-line FAQ > http://www.idomaincart.com/support/

Changes in Version 1.1:

- TLD search grouping
- Allow customer to set rules for Name Suggestion Wizard
- Enhanced discount pricing per item
- Transfer domain(s) option
- Hosting only option

1.1 Overview

  1. The user logs onto a search page powered by iHostingCart.
  2. Domain Availability Search: User enters a domain name and selects the TLDs to be searched. The TLD list and search function is configurable.
  3. Domain Availability Results: User views results of search (including WHOIS info, generated by 'rawwhois.cgi', if domain is taken) and can choose to add the name to their shopping cart. Each time a domain is added to the shopping cart, it's added to a cookie.
  4. Domain Shopping Cart: The shopping cart reads the cookie and shows a list of domains selected. These elements are dynamically generated by 'ihosting.cgi', within the same formatted template. The user then presses 'Proceed' button to continue with ordering.
  5. Hosting Shopping Cart: On the following page, the user can select the registration options you offer: For instance, the number of years domain to be registered, hosting, domain forwarding, e-mail forwarding, parking options and so on. When user clicks on the 'calculate' button, the script calculates the prices of each domain based on the user's selection. It also displays the total cost of the order as well as tax added. Both the prices of domains, hosting options and any other service you offer, and the tax percentage, are configurable.
  6. Order Processing: When user presses 'Proceed to secure order form' button, the contents of their shopping cart is passed onto 'order.cgi', which in turn prints out a previously formatted order form template. This order form page prints the pricing information as well previously specified form fields. These fields gather user information, which are then passed onto you or your automated script, along with hidden data (such as the list of domains to be registered, hosting options and pricing information), once the order is submitted.
  7. Order Confirmation: After order submission, the customer can be directed to a confirmation page.

1.2 Before You Begin

  • Make sure you have a text editor hat doesn't distort UNIX line feeds (ie. do not use Windows Notepad) so you can edit .idc files correctly
  • Decide where you're going to be uploading iHostingCart's files to and note the URL and path for this chosen folder
  • Make sure CGI files are executable in your chosen iHostingCart folder
  • Determine whether if your secure pages fall into one of the following categories: i) your secure pages are on a server provided by a automatic credit card processing provider ii) on your own secure server iii) on a different secure server (whereby the URL to your secure pages is not the same as the URL to your non-secure pages). Thinking about which category your secure pages fall into at this stage will make the rest of this README file more coherent.

3. INSTALLATION [Top]

3.1 Quick Guide

a) Move all the files (except 'license.txt' and 'README.htm') to your cgi-bin folder or a folder where .cgi files are executable

b) In that folder, set permissions on the files: chmod 755 *.cgi, chmod 744 *.idc, chmod 766 errorlog.idc

c) Using a text editor, open 'config.idc'. Set the URL of 'order.cgi' and 'ihosting.cgi'. Save it to the server.

d) Go to 'ihosting.cgi' in your web browser. If a domain search form appears, congratulations - you've installed iHostingCart

e) You can now customise the script. First, select the top level domains (TLDs) you want to offer for sale - just uncomment their lines in 'tldinfo.idc'

f) Then, set prices and hosting options in 'prices.idc'

 g) Check that your new options are working right

h) Finally, if possible move 'errorlog.idc' to a non-web folder, and set the path to it in 'config.idc'

3.2 Full Guide

a) First, move all the files (except 'license.txt' and 'README.htm') to a folder where .cgi scripts can be executed (this will normally be your cgi-bin folder, but ask your web osting provider if you are not sure.)

b) Now, set permissions on the files. The cgi scripts (ending .cgi) should be readable and executable via the web.

c) The files ending .idc just need to be world readable. On Unix systems, the commands are:
chmod 755 *.cgi
chmod 744 *.idc

d) Your error log needs to be world-writable, but not world-executable: on Unix, chmod 766 errorlog.idc On NT, permissions are set by folder rather than by file.

e) Because your error log is world-writable, you may wish to put it in a non-web accessible folder. Just alter the path_to_error_log variable to point to its place on your server. E.g. if you put it in your home directory, /home/jo, set "path_to_error_log = /home/jo/errorlog.idc".

f) You don't have to have all the files in your cgi-bin: only the scripts and the 'config.idc' file. The rest can be anywhere on your system, so long as you change the "path_to_" variables in 'config.idc' to point to them.

g) The "path_to_" variables require a path on your file system (either absolute, or relative to your scripts), not a URL.

h) You can move 'config.idc' if you really want to - you'll need to edit 'order.cgi' and 'ihosting.cgi', and change the variable $config_file_name, e.g. $config_file_name='/home/jo/config.idc'

i) Now start your web browser and check to see if the script is working by going to its URL.
- if you get an "internal server error" (error 500) result: check that you have set the correct permissions on the scripts. If that doesn't solve your problem, check your server error log (on Unix systems, usually found in /var/log/httpd) to see the cause of the problem, then go to http://www.idomaincart.com/support/ check out our FAQ and support forum.
- if you get an error message from the script itself: check the 'errorlog.idc' for more information. It may tell you which file can't be found. If so, check that file is in place and has permissions set correctly. If not, check all the files to make sure they are in place and have the right permissions set. If that doesn't solve your problem, check your server error log , then go to http://www.idomaincart.com/support/

j) Once the script is working, you need to customise it to suit your set up.

k) Choose which top level domains you want to offer to your customers by uncommenting the relevant lines in 'tldinfo.idc' - just remove the # at the start of the line. (To stop offering a top level domain, you put the # back at the start.)

l) Choose hosting options and prices in 'prices.idc'. If you are happy with the default hosting options, you can just edit the prices on the right hand side of 'prices.idc'. Otherwise, see the section on the hosting options menu for more detailed info.

m) Edit 'details.htm' to send the information from the script to yourself for manual processing, or to an external credit card processor for automatic processing

n) Customise your search form to look just the way you want it by editing 'search.idc'

o) Set up the Alternative Name Wizard to search for names on a particular theme

p) Blend iHostingCart with the look of your site by editing the HTML templates 'search.htm', 'order.htm' and 'details.htm' You can use an HTML WYSIWYG editor for this. See the section on templates and placeholders.

q) Customise error messages, form buttons and HTML even more by editing 'config.idc'

r) Choose a currency symbol and optionally a tax rate by editing 'config.idc'

4. CUSTOMISATION [Top]

iHostingCart uses powerful, yet simple, templates for ultimate customisation. You have complete control over the appearance of your pages.

4.1 Set Default TLDs
Normally you can control which top level domains are looked up by using the tlds_to_check value. But if this value is not sent (maybe a customer forgot to select a TLD from your checkboxes or dropdown list) the default value of default_tlds in 'config.idc' will be used This is a comma-separated list of TLDs.

4.2 Select TLDs
If you want a particular Top Level Domain to be searchable, you have to uncomment it in 'tldinfo.idc': just remove the # from the start of the line. [UPGRADE NOTE:] You don't need to include a price any more - this is handled separately by 'prices.idc'

4.3 Customising Searching
The file 'search.idc' lets you customise the input form to search for domains on the script page. This form will be printed in place of the **INPUTFORM** placeholder on your search template. Some extra HTML will be added to make sure that shopping cart contents are carried over to the search results page, even when cookies are not turned off. (That's why you can't just write the HTML for the form into the template page.) You can include anything that could normally go inside a form tag (except that the '#' sign is used to mark off a comment, so don't do "<font color='#FF0000'>"). Several different styles are included for you to try - if one of them suits you, just comment it out. The form should include at least one input named "domain_root" and (normally) one or more named "tlds_to_check" - see above for an explanation of how these work. Of course, you don't have to limit yourself to the search form on the script page itself. You can put search forms anywhere on your site. Just include an input named "domain_root" and one or more named "tlds_to_check".

4.4 Bulk Domain Searching

i) iHostingCart lets you look up multiple domain roots simultaneously. This powerful feature lets customers search for:
- the same domain name in different TLDs (yahoo.com, yahoo.net, yahoo.co.uk etc.)
- different domain names (yahoo.com, hotmail.com, foobar.com)
- a combination of domain roots and TLDs (yahoo, hotmail, .com, .net => yahoo.com, yahoo.net, hotmail.com, hotmail.net)

ii) Different several ways to send domain roots:
a) The basic way to make the script perform a lookup is to send it two pieces of information: domain_root and tlds_to_check. For example, sending the information domain_root=yahoo&tlds_to_check=.com will make it look up yahoo.com. You can send the information by the GET or POST method (although we recommend the latter). You can send the script multiple domain roots - perhaps from multiple text inputs: domain_root=yahoo&domain_root=hotmail&tlds_to_check=.com
This will look up yahoo.com and hotmail.com.

b) You can also send the script multiple TLDs:
domain_root=yahoo&tlds_to_check=.com&tlds_to_check=.net This will look up yahoo.com and yahoo.net.

c) Another way to look up multiple domain roots is to send a list of words in the domain_root key. The list can be separated by commas, spaces, carriage returns or semi-colons. Suppose you have the following HTML in your input form:
<textarea name='domain_root'></textarea><input type='hidden' name='tlds_to_check' value='.com'>
and a customer enters:
------------------------
yahoo; hotmail, internet
fortunecity,
amazon
------------------------
then the script will look up yahoo.com, hotmail.com, internet.com, fortunecity.com and amazon.com.
It's very hard for the customer to get it wrong! The script is even more flexible. If the domain_root contains names which already have a top level domain, then those domains will be looked up without adding the tlds in tlds_to_check.
Suppose the customer now enters
------------------------
yahoo; hotmail, internet
fortunecity.net,
amazon.co.uk
------------------------
The script will look up yahoo.com, hotmail.com and internet.com (because of the hidden input tlds_to_check=.com)
fortunecity.net and amazon.co.uk. It won't look up amazon.com or fortunecity.com.

4.5 Other Domain Search Methods

You may wish to have a domain search box on your front page or elsewhere on your site (or even on each page). There are a number of ways you can allow your customers to search for available domains:

a) Global Search Box
You can place the global search anywhere on your site. Place this code where you wish the global search box to appear, on as many pages as you wish. The '.com' and '.net.au' are just examples. Include as many TLDs you wish in the same format. iHostingCart will only search TLDs you specify. The search results will appear on iHostingCart's template page.

<form action="http://<your domain>/<iHostingCart folder>/ihosting.cgi" method="post">
<input type="text" name="domain_root">
<!-- Add as many TLDs below as you wish. Create a new line for each TLD in the same format-->
<input type="hidden" name="tlds_to_check" value=".com">
<input type="hidden" name="tlds_to_check" value=".net.au">
<!-- End of TLDs to be searched -->
<input type="submit" value=" Go! " name="submit">
</form>

Users can search for either a 'name' or 'name.com' using the same search box. For instance, if a user searches for 'mynewdomain', iHostingCart will list all available TLDs. If they search for 'mynewdomain.com', iHostingCart will show if 'mynewdomain.com' is taken or not. In addition, it automatically lists other available TLDs for 'mynewdomain'.

b) Tick Box TLD Selection
Copy and paste the following code in the place you wish the tick boxes to appear. Delete or add the list of TLDs as necessary, and change the form "action" to point at your 'ihosting.cgi' script.

<form action="http://<your domain>/<iHostingCart folder>/ihosting.cgi" method="post">
<input type="checkbox" name="tlds_to_check" value=".com">.com
<input type="checkbox" name='tlds_to_check' value=".net">.net
<input type="checkbox" name='tlds_to_check' value=".org">.org
<input type="checkbox" name='tlds_to_check' value=".com.au">.com.au
</form>

c) Drop Down Menu TLD Selection
Copy and paste the following code in the place you wish the search box and drop down menu to appear. Delete or add the list of TLDs as necessary, and change the form "action" to point at your 'ihosting.cgi' script.

<form method="post" name="whois" action=" http://<Your domain>/<iHostingCart folder>/ihosting..cgi">
<strong>www.</strong><input type="text" size="20" maxlength="64" name="domain_root"> <strong>.</strong> 
<select name="tlds_to_check" size="-1">
<option selected value="">Select</option>
<option value="">------</option>
<option value=".com">com</option>
<option value=".net">net</option>
<option value=".org">org</option>
<option value="">------</option>
<option value=".co.uk">co.uk</option>
<option value=".ltd.uk">ltd.uk</option>
<option value=".org.uk">org.uk</option>
<option value=".plc.uk">plc.uk</option>
<option value="">------</option>
<option value=".uk.co">uk.co</option>
</select> 
<input type="submit" value="Search">
</form>

d) Group Searching
You can also send the TLD list as a comma separated list:

<input type="hidden" name="tlds_to_check" value=".com,.net,.org">

NOTE: This makes it easier to group tlds in a drop-down box e.g.

<select name="tlds_to_check">
<option value=".com,.net,.org">Traditional domains
<option value=".shop,.biz,.pro">New domains
</select>

4.6 Name Suggestion Wizard

iHostingCart includes an alternative name suggestion wizard. This gives your customers a simple way to look up alternative domain names. They do this by selecting one or more themes, along with a root word. The script then creates domain names by adding words from the themes to the root word. For example, if a customer entered the root word "repairs" and the theme "speed", the script would look up "speedyrepairs","kwikrepairs","fastrepairs" and so forth. You can also set options to look up hyphenated names (like "kwik-repairs" and "speedy-repairs") as well as names with the theme after the root word (like "repairsfast" and "repairskwik").

a) Choosing a Theme to Look Up
The default themes included are in 'config.idc'. They can be chosen by sending the script a name and value as follows:
lookup_theme=<selected theme>
For example, lookup_theme=theme_speed will use the speed theme.

b) Setting Lookup Options
'config.idc' allows you to choose whether to look up hyphenated themes: just set the value of try_with_hyphen to 1. It also lets you choose whether to look up names where the theme word comes after the root word: just set the value of try_after_root to 1. If both these options are set to 1, their effect will be combined:
you'll get "fastrepairs","fast-repairs","repairsfast" and "repairs-fast"!

c) Creating New Themes
Creating new themes is easy - just add a line to your config.idc file. For example, to create a new "fishing" theme, add:
theme_fishing = fish, bait, angling, trout
You can then use this theme in the normal way by sending the script "lookup_theme=theme_fishing"

f) A Warning!
If you have ten words in a theme, the script will look up ten domain names for each root word you pass it. With try_with_hyphen and try_after_root turned on,
it'll create forty domain names. If you then pass the script ten root words (say because a customer enters ten words into a TEXTAREA field), the script will have to look up 100 or even 400 domains! If there are multiple top level domains to look up, this will make even more lookups. This can seriously slow your script down as it waits for 400 responses from WHOIS servers. It is probably better not to combine a TEXTAREA field with a theme selector.

g) Allow Customer to Define Rules
You can override the default settings for try_with_hyphen and try_after_root defined in config.idc by sending the following values as parameters. This means your users can decide whether to use hyphens or not. In your search template, you would insert the following in between the <form ...> action that sends the search request to ihosting.cgi:

<select name="lookup_theme">
<option value="speed"> with a Speed theme
<option value="b2b"> with an e-Business theme
</select>
<br>
<input type="checkbox" name="try_with_hyphen" value="1">Use hyphens with your themes

4.7 Creating Template Pages

Some example templates have been included in the zip file. Take a look at them to see how templates work. You may find it more convenient to use one of these existing files and simply adding your own header, footer, images etc. where necessary. Alternatively, you can create more sophisticated templates of your own from scratch.

To create a template, you can use an HTML editor to write up raw code, or use your favourite WYSIWYG editor like Dreamweaver, Frontpage, or Homesite. Don't put in any input forms or checkboxes - just design the look, feel and layout of the page by adding your own headers, footers, images, tables and so on. When you are satisfied, you will need to add "placeholders" to your page. These will be replaced dynamically with HTML content by the script.

To create a placeholder within a template, just type two asterisks (**) followed by the placeholder name, then two more asterisks (**) in your raw code/page design. Note: placeholders are case sensitive ie. they should all be capitalized. A placeholder is dynamically replaced with the scripts contents when the script is run.

You don't have to worry about what order placeholders should go in. There are three templates containing placeholders: 1. Search template: contains the initial search page function 2. Order template: lets customers select hosting and registration options for each domain 3. Details template: takes personal details, while the order.htm template .

1. Search Template
There are five placeholders available for the search template, printed out by 'ihosting.cgi':

**ERRORMSG** --> prints any error messages
**INPUTFORM** --> prints the search form from search.idc
**RESULTS** --> prints any search results. If domains are available, prints a button to add them to the shopping cart.
**SHOPPINGCART** --> prints the contents of the shopping cart, if any, with a button to remove domains
**CHECKOUT** --> prints a button to proceed to the order form ('order.cgi')

2. Order Template
Our order form script sets the industry standard for power and flexibility. It allows you to offer your customers a menu of options for each domain they register: it automatically recalculates price information and can check for errors in customer input. The order template, pointed to by path_to_order_template in 'config.idc', displays options for each domain. When correct options have been selected, the customer can move on to the details template, which lets you ask for personal details like name, address and credit card number. The order template for 'order.cgi' has five placeholders available:

**ERRORMSG** --> prints any error messages
**SHOPPINGCART** --> prints out each domain in the shopping cart, with a menu of options derived from 'form.idc', plus the price and any error messages if options have been selected
**CONTINUE** --> if options have been selected and there are no errors in user input, prints a button to move on to the perrsonal details entry form('details.htm')
**TOTALPRICE** --> prints the total price of selected options
**TAXEDPRICE** --> prints the total price of selected options, including tax (or just the total price if "tax" is not given a value in 'config.idc')

3. Details Template
'details.htm' can include a form to send customer's details (as well as the per-domain options previously selected) on to a script of your choice - an external credit card processor, an automatic domain registration backend, a mailing script like formmail or SimpleSecure, or your company's custom script. The details template shows the final stage before an order is submitted. It allows you to print a form to collect information from the user, such as name, address and credit card details. [UPGRADE NOTE:] This is the equivalent of the old 'order.htm' template and uses much the same placeholders. Six placeholders are available:

**ERRORMSG** --> prints any error messages
**LISTINPUT** --> prints a comma-separated list of domains ordered.
**ENCINPUT** --> prints a list of domains ordered, encoded for passing in a URL (i.e. separated by "+")
**HIDDENINPUTS** --> this prints a list of _all_ the options chosen on the previous order.htm page, as a series of <input type='hidden'>s. You can customise the appearance of these INPUTS using the variables hidden_input_suffix, domain_input_name and price_input_name in 'config.idc' - see that file for more details
**TOTALPRICE** --> prints the total price of selected options
**TAXEDPRICE** --> prints the total price of selected options, including tax (or just the total price if "tax" is not given a value in 'config.idc')

4.8 Font Styles

You can generally specify how text appears by entering <FONT ...> tags in appropriate configuration files and in templates. For example, if you wished to change the font of the domain results section (i.e. the text which prints whether a domain name is taken or not) you could do the following in 'config.idc':
html_before_each_domain = <font face="Verdana, Arial, Helvetica, sans-serif" size="2"><b>
html_after_each_domain = </b></font>
This will print each available or taken domain in Verdana font, size 2 and bold.

5. DEFINE HOSTING OPTIONS [Top]

iHostingCart's most powerful, long-awaited and advanced feature is that it lets you offer different hosting options to customers. Instead of just registering a domain, you can now ask them how long they want to register a domain for, offer them a menu of hosting options, or ask for any per-domain information you want. To set this feature up, you will need to edit two files: 'form.idc' and 'prices.idc'.

'form.idc' contains the HTML snippet you want to show the customer for each domain in the shopping cart.
'prices.idc' lets you interpret the results of the selections the customer makes, to print prices or error messages.

5.1 Introduction
To get an idea of how these two files work, take a look inside 'form.idc' and 'prices.idc'. By default, the first section is uncommented
-1-Simple selection of registration--- ...

Try out the script by searching for some domains and going to the order form with one or two in your shopping basket. See what gets printed out. Try selecting different options and see how the prices change. With a non-US top level domain, try selecting "10 years". You should get an error message warning you that non-US domains can only be registered for two years. Now, in both 'form.idc' and 'prices.idc', comment out section 1 by adding "#" to the start of every line. Then, uncomment section 2 in both these files, by removing the "#" at the start of those lines. Save the files. Now go back and search for some different domains, then go on to the order form. It should show you a different menu of options. Again, try different options to see what prices result. See what happens if you type in an invalid email address. By now you probably want to know how it works....

5.2 Create Options
'form.idc' contains some HTML. This will be printed for each domain in the shopping basket. This will probably include some <inputs>, because the whole shopping basket is enclosed in <form> tags. The <inputs> can be used to show customers a menu of options - take a look at some of the default options in'form.idc'. The HTML in 'form.idc' is not printed exactly as you write it. The script has to parse it and add a suffix to every input value so that it can distinguish between the options selected for each domain. (The suffix consists of three underscores followed by a number: ___1, ___2, etc., where each number corresponds to one domain in the shopping basket.) To make it easy for the script to parse your HTML, please:
- don't use the characters < and > except to mark off HTML tags: e.g. don't do this:
<input type='submit' value='>>>Go>>>'>
- As with all the configuration files, don't use the hash '#' in HTML (e.g. <font color='#FFFFFF'>) because the script will interpret it as a comment and ignore the rest of the line.

5.3 Set Option Prices
When a customer hits the "calculate prices" button, the script will receive the inputs he or she has chosen. It then has to know what all these inputs mean, so it can work out a price for the options for each domain, and print any error messages. To tell it, you use the 'prices.idc' file. This file works very simply. It consists of two sides, divided by a "|". On the left side is a series of key-value pairs, like:
years=1
or:
hosting = full_hosting, fp_extensions = 1
On the right side is a result, like:
15.00
or
ERROR Please enter a valid email address

For each domain entered, the script reads down the lines in the file. On each line, if the keys and values on the left match the inputs chosen by the customer for that domain, then the result on the right is used. For example, suppose you are using the first default set of options, underneath #-1-Simple selection of registration--- Suppose a customer has two domains in her cart: foo.com and bar.net. For foo.com, she chooses a registration length of two years. For bar.net she chooses one year. When she hits the "calculate prices" button, these options are sent to the script. The script reads through the file for foo.com, and gets to years=1 but for foo.com, the customer has chosen "two years", so it ignores this line. When it gets to years=2 then it looks to the right side of the line and prints out 20.00. Then it reads through the file for bar.net. When it gets to the same line, it discovers that bar.net has an option of "years=1" associated with it. So it looks at the right side of the line for the result. In this case, that result is: 10.00 - set the price of this domain to 10.00.

5.4 Prices and Warning Messages
Results consist of a number and/or an ERROR or WARNing. If the result is just a number, this number will be added to the price of the particular domain. Suppose you want to charge $10 extra for offering hosting an NT server to a customer. You would have some suitable HTML in your 'form.idc' file: maybe
<input type='checkbox' name='NT_hosting' value='1'> Tick here to select NT webhosting

Then you would have this line in your 'prices.idc' file:
NT_hosting=1 | +10.00

For each domain where the checkbox was ticked, the price of that domain would go up by $10. If the result has an ERROR in it, then:
- the price for that domain will be set to zero
- anything after the ERROR will be printed as an error message for that domain
- the CONTINUE button to move on to the next part of the form will not be printed

Suppose you didn't like Windows NT for some reason. You could add this line to 'prices.idc':
NT_hosting=1 | ERROR <b>Are you crazy? Choose a decent operating system!</b>

As this example shows, you can include HTML in the error message, although as usual, don't use the "#" sign because the script will think it's a comment. Perhaps you're not so strongly against NT as all that. You just want to warn people about NT, not to forbid them from choosing the option. In that case, you can use WARN, like this, to print a warning.
NT_hosting=1 | +10.00 WARN <b>Well, it's your choice... good luck...</b>

This will still print the price, with the extra 10.00, and will let users move on to the next page to enter their personal details. Anything after the WARN will just be printed as an advisory note. You can use this to give information about the option selected, e.g.:
email_fwd=1 | 5.00 WARN Email forwarding may take a day or two to come into effect.

5.5 Advanced Pricing Options
You can use a number with a plus or minus sign to add to, or deduct from, the total price of a domain. For example:
hosting_options=gold | +15.00 # Gold hosting package costs 15.00
*ITEMS=*GT 10 | -1.00 # 1.00 off all domains if you buy more than ten domains

If you use a number without a plus or minus sign, the price will be reset to that number. All previous calculations will be disregarded (although you can change the price by putting more lines lower down in prices.idc. For example:
*TLD=.com | 20.00
*TLD=.net | 20.00
*TLD=.org | 20.00
years=1 | -10.00
years=5| +30.00
years=10 | +80.00 # 10.00 per domain per year for .com domains
hosting_options=platinum, *TLD=.com | 0.00 # Buy a Platinum hosting package and get a FREE .com domain
Price is now set to 0
hosting_options=platinum | +250.00 # The hosting package costs 250.00
Price for a .com domain and platinum hosting is 250.00 exactly

5.6 Special Keys and Values
Sometimes you want to do more than check for a specific value. You might want to check whether the customer has entered a valid email address, for example, or a number. iHostingCart lets you use special keywords to perform these checks. These keywords consist of an asterisk ("*") followed by capital letters.

1. Special Values

*ANY
Checks whether a value for the key has been inputted at all. E.g.
years=*ANY checks whether any value has been entered for years.

*NUMBER
Checks whether the value input is a valid number (with or without a decimal point).

*EMAIL
Checks whether the value input is a valid email address. Note that this checking is not perfect (conforming to the official specification for email addresses would take a lot of code!) but it should make sure that something at least vaguely like an email address has been input.

*WWW
Checks whether the value input is a valid web address. Again, this checking is not perfect (far from it!) but should catch the most serious errors.

*NOT
At the start of any value, this checks that the value is NOT what follows. E.g.
email_fwd = *NOT *EMAIL
checks that email_fwd is not an email address (and would probably print an ERROR as its result).

years=*NOT 2
checks that 2 was not chosen by the customer as a value for the "years" input.

*GT, *LT
*GT and *LT check that the chosen input is Greater Than or Less Than the following value. E.g.

years = *GT 5
checks that a numeric value of more than 5 was chosen for the "years" input.

2. Special Keys

*TLD
*TLD checks the top level domain of the domain name being checked. You can use this to set prices:
*TLD = .com | 20.00
*TLD = .co.uk | 15.00
would give a basic price of 20.00 for a .com domain, and 15.00 for a .co.uk domain.

*ITEMS
*ITEMS checks the number of items in the basket as a whole. You can use this to offer bulk discounts:
*ITEMS = *GT 5 | -5.00
gives 5.00 off the price of each domain if there are more than 5 domains in the shopping basket. Note that this will apply to all domains - not just to the 6th, 7th, 8th et cetera.

*ITEMNUMBER
gives the number of the item in the shopping basket. For example
*ITEMNUMBER = *GT 3 | -10.00
gives a 10.00 discount for all domains after the first 3. (Whereas
*ITEMS = *GT 3 | - 10.00
would give a 10.00 discount on all domains if there were more than three domains in the shopping cart.

*INTEGER
checks if an input is a whole number

5.7 Domain Transfer Options
You can use order.cgi to transfer domains, if someone wishes to transfer a domain which they already own to your hosting company. Set up a page with your site design etc. and include a form like this:

<form action="/path/to/order.cgi" method=post>
Enter your domain to transfer:<input type="text" name="transfer">
<input type="submit" value="Transfer Domain">
</form>

You can also let people enter multiple domains to transfer, separated by a comma, semi-colon or space. So you could replace the text input with a text box, like this:

<textarea name="transfer">Enter a list of domains to transfer here</textarea>

You will then need to set up your prices.idc file to differentiate between transferred domains and new domains - obviously, nobody wants to pay for registering a domain they already own. You can do this using the key "transfer" to determine if a domain has been transferred.

Here is a sample prices.idc file using "transfer"

*TLD=.com | 10.00
*TLD=.net | 10.00
*TLD=.org | 10.00
# any other tlds and prices for registration could go here

transfer=1 | 0.00 # If the domain is being transferred, the price will now be set back to zero

hosting_options=basic | +25.00 # these hosting options will still apply even if the domain is being transferred
hosting_options=silver | +40.00
hosting_options=gold | +75.00

5.8 Hosting Only Options
This version of iHostingCart was not designed to be a 'hosting only' shopping cart, however, we have tried to include a facility to take customers directly to the hosting options page using the 'domain transfer' work around. Create a normal page with your site design etc. and include your hosting plan details. You would then create a 'order a hosting plan' button on your page like this:

<form action="/path/to/order.cgi" method=post>
<input type="hidden" name="transfer" value="mydomain.com">
<input type="submit" value="Order hosting!">
</form>

Place this button wherever relevant. Remember, you can make the buttons look nicer by turning them into images ...

<input type="image" src="/path/to/images/go_to_hosting_plan.gif" border="0" >

This will then take your customer directly to the hosting options page where they can select your hosting plans.

6. ORDER PROCESSING

6.1. Introduction

iHostingCart lets you choose what to do with the information gathered by 'order.cgi'. When correct options have been selected for each domain, the customer may continue to the final part of the order form. The form then prints out the details template (selected by path_to_details_template in 'config.idc'). On this template you should design a form to gather all the information you need from the customer and send it on to your chosen destination - perhaps an automatic registration script, a credit card processor, a form mailing script like formmail, or your own in-house solution. In order to include all the information gathered on the previous page - the options chosen for each domain - you need to include the **HIDDENINPUTS** placeholder in the 'details.htm' template. This will then print out all the chosen options as a list of <input type='hidden'>s.

You may need to customise the names of these inputs. For example, the register.com automatic registration script requires you to send it
a list of domains as follows: Domain1=foo.com, Domain2=bar.net etc.

6.2. Customisation

You can customise inputs using the "order form customisation" section at the end of 'config.idc'. This has three values:

1. hidden_input_suffix
"hidden_input_suffix" is printed out after each input specified. The special value *NUMBER  is replaced by a number indicating which group each option belongs to. 
For example, if your form has inputs called "email_fwd" and "hosting_option", hidden_input_suffix has a value of "-*NUMBER", and the customer has ordered two domains, foo.com and bar.com, **HIDDENINPUTS** will print out:
<input type='hidden' name='email_fwd-1' value='[whatever option the customer selected for foo.com]'>
<input type='hidden' name='hosting_option-1' value='[whatever option the customer selected for foo.com]'>
<input type='hidden' name='email_fwd-2' value='[whatever option the customer selected for bar.com]'>
<input type='hidden' name='hosting_option-2' value='[whatever option the customer selected for bar.com]'>

2. domain_input_name
"domain_input_name" selects the name of the input which contains the domain name purchased. By default this is "domain" but if you select something else, that will be used instead. So in the above case, normally **HIDDENINPUTS** would print
<input type='hidden' name='domain_1' value='foo.com'>
<input type='hidden' name='domain_2' value='bar.com'>
but by selecting domain_input_name = name2register, you could get
<input type='hidden' name='name2register_1' value='foo.com'> etc.

3. price_input_name
"price_input_name" selects the name of the input which contains the price for each individual domain purchased (and any associated hosting options). This is not the same as the total price for all domains, which can be passed quite simply, using e.g. <input type='hidden' name='totalprice' value='**TOTALPRICE**'>

6.3. Order Processing Using Secure Servers

There are two ways to use a secure server to process orders with iHostingCart.

1. Install on secure server

If you have available a secure server where you can install CGI scripts, you can install 'order.cgi' on that server.

You will need to place a copy of 'config.idc' in the same folder where 'order.cgi' is installed, and alter the path_to_ variables in 'config.idc' to point at the various config files and templates used by iHostingCart. If your secure server is on a separate file system from your normal server, then you will also need to provide copies of the following config files on the secure server's file system: 'order.htm', 'details.htm', 'form.idc' and 'prices.idc'. (Note: you cannot set the path_to_ variables in 'config.idc' to be URLs: they have to be paths on an individual file system.)

2. Send to secure server

If you can't install CGI scripts on your secure server, you can still set up 'details.htm' to point to your secure server. It is perfectly safe to collect customer details such as credit card numbers on 'details.htm', so long as they are then transmitted by a secure link to a 'https://' URL. So just create a template with a form like this:

<form action = 'https://my.secureserver.com/cgi-bin/myscript' method='post'>...

6.4. Automated Credit Card Processing

iHostingCart is easy to integrate with automated credit card processing scripts. There are two stages to do this:

1. Make sure that domain hosting options in form.idc will be ignored by your credit card processors.

For example, if you use Worldpay to process credit cards, they require that all input names except those used by their credit card processing scripts begin with 'M_' or 'C_'. So you would want to set up form.idc with options like:

<input type='checkbox' name='M_hosting_package' value='email_forwarding'>

instead of:

<input type='checkbox' name='hosting_package' value='email_forwarding'>

This will ensure that these options are ignored by your credit card processing script. Other credit card processors may do things differently, so consult your documentation to find out how yours works.

2. Set up 'details.htm' to create a form with the inputs your credit card processors need, and point it at the credit card processing script.

For example, Worldpay requires a field named 'instId' with your Worldpay installation ID, a field named 'price' which gives the total price to charge to a customer's card, and a field named 'currency' which gives the chosen currency to charge in. So you would set up 'details.htm' to contain a form like this:

<form action = "https://select.worldpay.com/wcc/purchase">
<input type="hidden" name="instId" value="[your Worldpay installation ID]">
<input type="hidden" name="price" value="**TAXEDPRICE**">
<input type="hidden" name="currency" value="USD">
**HIDDENINPUTS**
[ ... other fields of information required by you or Worldpay ... ]
</form>

Here, **TAXEDPRICE** will be replaced by the total price of all the options selected by the customer, including tax. **HIDDENINPUTS** will be replaced by a list of all the options selected by the customer, each formatted as an <input type='hidden' ...>. These options will be passed to the credit card processing script, and, if you have set up 'form.idc' correctly in stage 1, will be ignored but passed on to you, either as part of a receipt sent to you after the order is processed, or as a callback which passes the options on to a URL of your choosing. (This enables you to continue processing the options automatically after credit card payment, perhaps by registering the domains using a bulk registration script. Consult your credit card processor documentation to see if this option is available to you.)

6.5. Order Confirmation Page

If you wish to have a confirmation page shown to the customer, you will need to set up the script pointed to by 'details.htm' to print out this page, or redirect the customer to it. With formmail, for example, you would include a hidden input as follows:

<form action='http://my.server.com/cgi-bin/formmail.pl' method='post'>

<input type='hidden' name='redirect' value='http://my.server.com/thankyou.htm'>

[ ... other inputs and placeholders as required ... ]

</form>

Other scripts may require setting up differently, or your credit card processor may allow you to configure the credit card processing script to return customers to a URL on your site after a successful transaction.

8. SECURITY [Top]

Your iHostingCart script should be completely secure. There are no known security holes. This section exists to help you make doubly sure that security is watertight.

- error log file
The error log file "errorlog.idc" must be writable by the PERL script. The ideal way to do this is to make it owned by your server. Running Apache, this can be done by:

chown nobody:nobody errorlog.idc

if Apache's username is "nobody". Then make it writable only by apache and readable by others (chmod 744). However, you may not be able to change file ownership on your server, as this normally requires root access. The next best alternative is to make sure that the error log is in a non-web-readable file, and to make it world writable but not executable (chmod 766). For example, if your cgi-bin is at /home/httpd/cgi-bin, then you could put the error log in /home/<your home directory> and change the value of "errorlog" in 'config.idc' to

errorlog = ../../<your home directory>/errorlog.idc

If you absolutely do not want a world writable file on your system at all, then you can just remove the error log and leave errorlog blank:

errorlog =

This will not affect how the script works.

- config files and templates
The config files and templates can be made world readable but not world writable (chmod 744). They should not therefore pose a security risk. If you want to avoid anyone being able to look at them, just put them in a non-web-readable directory and change the relevant values in 'config.idc'. 'config.idc' itself ought to be in the same directory as the scripts, but if you are feeling really paranoid you could change the early lines of 'ihomain.cgi', 'order.cgi' and 'confirm.cgi' to point to a non-web-readable directory and put 'config.idc' there.
