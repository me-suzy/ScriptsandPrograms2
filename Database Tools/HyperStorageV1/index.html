  • About HyperStorage

    First off, it's freeware, so if you can't get it to work the way you want, you are probably doing something that's more complex than just million row data libraries, or you didn't set it up correctly. Remember that the purpose of this program is to prepare for large data build up, for your future online business or forums; this engine will prepare you.

    Once you learn how powerful the technology is, you can extend the power by interacting with the coding which drives the engine. The coding is extremely short, but the technology & idea is extremely extensive. It's like having the full power of MySQL in one script. Although, this isn't a magical 'dream-machine' code, it does have it's limits, and you must code carefully.

    What's the purpose of this engine?

    This engine will make data storage & data locating much faster for your scripts and MySQL. Just remember to keep your KEY table as small as possible, which is why we only have KEY, EXTRA, and ID in the key table. The Key table is sort of like a table which holds many different doorways. And with the doorway information, you can specifically locate where the Data you are looking for, is stored in your MySQL. When you are using different HyperTypes, new key tables are created. You do not want to combine your key tables, because the test limit for key tables is 180 million rows, with indexing, it causes a very bad performance issue. So try not to exceed 100 million key rows. Remember, the entire purpose of this engine is to keep your data in seperate tables, to make data returns & operations go much faster.

    Consider this Indexing for Tables, similar to Indexing for Columns and Rows, except this finds tables, where the key you are looking for is located.

    MySQL does not offer Indexing for Tables, which is why I built this engine!

    Power of Indexing & HyperStorage

    If you haven't learned about indexing, then there is a lot about mysql that you don't know. Let me explain. Lets say for example, you have 500,000 Users in a table, and the information stored in this table is stuff like User Name, Password, Email Address, Profile information, saved information, things that the user needs when they are logged into your site.

    If you haven't indexed this information, MySQL could take as long as 1 to 10 seconds to return with the needed data, depending on how many rows, and how much data is stored in your table.

    So to speed this up to say 0.005 seconds (extremely fast), you need to create an index on the column you use to find and return data. Remember to only index columns you use to find data (for example: SELECT firstname FROM users WHERE username='jimmysticks'). You would need username indexed in order to perform a fast search. Indexing makes searching go very fast, however indexing doesn't always go extremely fast, because you may take on so much data, regardless of indexing, your searches can slow down. And the entire reason this engine was built is to speed that up.

    Please note, if you are indexing a column on a large table, it could take as long as 20 minutes. So be patient when you performing any kind of indexing, or FULLTEXT alterations.

    Templates [Your table design]

    You must have templates for each different $type. When you make a location, you are creating a new place for data to be stored. If you were to make $type 'users', it would create a database called users (if it doesn't exist), and a new identifier (500) if it doesn't exist. So you could find table 500 in the database $type 'users', and this is where you would store your data. Automatically, HyperStorage engine will detect when table 500 is reaching it's 500,000 row limit (or whatever your limit is set to - see config.php to set a limit) , and create 501. This is what makes this engine so unique: it's ability to keep track of your data, and not let tables build up. SEE templates.php TO MAKE YOUR OWN TEMPLATES. SOME EXAMPLES OF ACTUAL TEMPLATES ARE USED. RECOMMENDED TO KEEP YOUR DEFAULT TO MyISAM and UTF8.

    Functions, Syntax, and Details

    include "locations.php";


       MakeLocation    ()
    Syntax
    $returnlocation = MakeLocation($type,$thekey,$extra[optional]);

    Returns

    Brings back a location of the created location. The engine will decide where to store the data, depending on certain factors specified in your config.php file. You use the $returnlocation to store your actual data. The engine takes care of making the necessary tables & databases and key locations; you on the otherhand are responsible to store / retrieve the actual data you put into these location tables, not the engine.

    Requirements

    TEMPLATES: See templates.php to make your templates.

    When you make a location, you are creating a new place for data to be stored. If you were to make $type 'users', it would create a database called users (if it doesn't exist), and a new identifier (500) if it doesn't exist. The table 500 has to be created under database $type 'users'. So where do you think it will get the actual table design? You have to make it, and put it into the templates.php file for use. If it doesn't exist, the table users.500 will never be made, and the operation will abort with a return 'no template'. You need to do this for any $type you make.

    Safeties

    The engine will not delete / overwrite existing databases, tables, or rows, it will only Create/Insert. If you experience issues with your templates, you must go and remove the key from location.$type table, or your MakeLocation will return "exists", causing you to have an aborted function. MakeLocation may also return "no template", "exists", "failure", etc, depending on the problem. You may need to add different return failures, however the included abort messages seem to be fine the way they are.

    Purpose

    Creates a new location for a $type, and creates the key inside of the $type table, and stores the key's new physical location in the ID column of your locations.$type table. The ID column tells you where the table is stored, for example: INSERT INTO $type.$returnlocation SET email='$thekey';". This would be proper usage of the "made" location. $type defines the data type, such as users, orders, forums, etc, and $thekey defines a specific piece of information used to find the data.

    Fields

    $type *     :     This is more important than the actual key. This is the type of system you are creating a location for. It can be ANYTHING. In our demo, we use it for 'users'. So if you wanted to make a location for 'users', you would specify syntax: $type='users';. You can store many different types of locations, 'users', 'orders', 'dvd', 'cd', 'products', 'mailinglist'. Try not to use spaces, MySQL may not be compatible with it. Any time you make a location with a $type='users', if you need to find a location, you must specify $type='users' as well, because it will not know where to find the location otherwise. Example of doing this is below.

    $thekey * : This is the key. Without a key, you can't find your data, and without your data, nothing happens. A key can be a specificly identifyable piece of information. For example, bobsmith@aol.com would be unique in this case, because you wouldn't want 500 bobsmith@aol.com emails in your databases, unless they were different types. For 'users', you would only want 1 bobsmith@aol.com, but you could also have 1 bobsmith@aol.com for other types, such as 'forums', 'wishlists', 'carts', etc. Make sure it's unique for that specific $type, because you don't want two of the same key in the same $type table, as mentioned in the Fields informaiton for $type. For the key, you should use an identifiable piece of information, associated with a row you intend to insert into the created location table. Example of doing this is below.

    $extra [optional] : This is an extra field, sometimes you may provide nicknames here, or a password, however, this field is only an extra storage tool used in MakeLocation. This field is OPTIONAL. You are not required to use this field when making a location, however it's there as an additional identification field, but it is not used with FindLocation() function.

       FindLocation()
    Syntax
    $mylocation = FindLocation($type,$thekey);

    Purpose

    Finds a location identifier of a key. Returns with the location, so you don't have to search through several tables to find it.

    Fields

    $type *     :     As explained in MakeLocation, this is the same $type you will use to find your key.

    $thekey * : As explained in MakeLocation, this is the same key you will use to find where your data is stored.


       An example is show below in the Examples section



       Examples of using the above functions

            # Pull back his information, so we can see what happened...
            # You can use my SmartQuery to pull back single data, or an Array, ARR before the command does arrays.

            $myemail = $myinfo[0]; # pulls back from array

           if ($myemail=="") {

Please verify that everything was created by viewing your MySQL Databases & Tables, using the MySQL command prompt or PHP Administration Tool<br>";
           
            } ELSE {
           
            # Data came back!
            # Success! The engine worked 100%!
           echo "This shows you that it worked correctly. '$mykey' in table '$mylocation' in database '$mytype' were created successfully. ";
           
            }

    EXAMPLE FOR FindLocation()


    <?

    include "locations.php"; # You will need this to make the functions below work

    $type = 'users'; # This is the database created to store your users.
    $mykey = 'bob@aol.com'; # This is the key, his unique identifier in 'users'.

    # Find where this person is by their E-Mail Address
    $mylocation = FindLocation($type,$mykey);

    if ($mylocation > 0) {
    # Found him...
    echo "Found him in '$type' database, in table '$mylocation'. You can access his data with mysql, or php script by doing this: <br><br>" . 'SELECT * FROM $type.$mylocation WHERE email="' . $mykey . '";<br><br>Congratulations!';
    } ELSE {
    # Not Found - He probably doesn't exist. Or he there was an error when you used MakeLocation.
    echo "Sorry, '$mykey' was not found in '$type'.";
    }

    ?>


    Run the above script and output will be generated.
